CSS Architecture (1) - Style Guide

2017-08-17

Why we talk about CSS Architecture?

Our current status.

  1. Large scale and code base, long running project.
  2. Several developers of differing specialities and abilities.

What issues or inconvenience I’ve had.

  1. Coding with uncertainty, worrying my code style is not readable and inconsistent.
  2. Painful while naming new variables.
  3. Several writing style across code base.
  4. Need a list telling which browsers needs to be supported, e.g. > IE10

Beneficial approaching for CSS code.

  1. Has a clear writing guideline and instruction to follow.
  2. Looks clean and tidy, so it would be easy to read and maintain.
  3. Comments in detail, it tells Devs things they need to be aware.

Further, it would be great to achieve these across codebase.

  1. Set the standard for code quality.
  2. Promote and keep consistency.
  3. Code will always look and feel familiar to all members.

General rules for CSS

Do

  1. Use soft tabs with two spaces.
  2. Put spaces after : in property declarations, e.g. color: red; instead of color:red;
  3. Put spaces before { in rule declarations, e.g. .modal { instead of .modal{
  4. Write your CSS one line per property.
  5. Add a line break after } closing rule declarations.
  6. When grouping selectors, keep individual selectors on a single line.
  7. Place closing braces } on a new line.
  8. Add a new line at the end of .less files.
  9. End all declarations with a semi-colon.
  10. Lowercase all hex values, e.g., #ffffff. for me lowercase letters are much easier to discern when scanning a document as they tend to have more unique shapes.
  11. Quote attribute values in selectors, e.g., input[type="text"]. They’re only optional in some cases, and it’s a good practice for consistency.
  12. Put spaces before and after child selector div > span instead of div>span.
  13. Use BEM as naming convention.(See below).
  14. Always use double quotes.

Don’ts

  1. Avoid using HTML tags in CSS selectors(with some exceptions like CSS resets).
  2. Don’t use ids in selectors, for example, #button is overly specific compared to .button, and is much harder to override, read more about the headaches associated with IDs in CSS here.
  3. Don’t nest more than 3 levels deep, because nesting selectors increases specificity, meaning that overriding any CSS set needs to be targeted with an even more specific selector. This quickly becomes a significant maintenance issue, also damage to selector performance. However, prefer using nesting as a convenience to extend the parent selector over targeting nested elements, or pseudo selectors and state selectors, e.g. nesting :hover, :focus, ::before, etc. is OK.
  4. Never ever use !important, it greatly increases the power of a CSS declaration, making it extremely tough to override in the future. It’s only possible to override with another !important declaration later in the cascade, harmful for maintaince.
  5. Avoid shorthand properties (unless you really need them), it can be tempting to use, for instance, background: #ffffff instead of background-color: #ffffff, but doing so overrides other values encapsulated by the shorthand property. (In this case, background-image and its associative properties are set to “none”. You can see more details here.
  6. Avoid specifying units for zero values, e.g., margin: 0; instead of margin: 0px;.
  7. Don’t include spaces after commas within rgb(), rgba(), hsl(), hsla(), or rect() values. This helps differentiate multiple color values (comma, no space) from multiple property values (comma with space). E.g. rgba(255,255,255,.8) instead of rgba(255, 255, 255, .8).
  8. Don’t prefix property values or color parameters with a leading zero (E.g., .5 instead of 0.5 and -.5px instead of -0.5px).

BEM

Block:
Unique, meaningful names for a logical unit of style. Avoid excessive shorthand.

  • Good: .alert-box or .recents-intro
  • Bad: .feature or .content

Element:
Styles that only apply to children of a block. Elements can also be blocks themselves. Class name is a concatenation of the block name, two underscores and the element name. Examples:

  • .alert-box__main
  • .expanding-section__section

Modifier:
Override or extend the base styles of a block or element with modifier styles. Class name is a concatenation of the block (or element) name, two hyphens and the modifier name. Examples:

  • .alert-box__main--active
  • .expanding-section__section--expanded

BEM Best practices

Let each classname take their responsibility.

  • Good: <div class="my-block my-block--modifier">
  • Bad: <div class="my-block--modifier">

Don’t create elements inside elements. If you find yourself needing this, consider converting your element into a block.

  • Bad: .alert-box__close__btn

Rule Ordering

Properties and nested declarations should appear in the following order, with line breaks between groups:

  1. Rule declarations.

  2. Grouping mixins at the end makes it easier to read the entire selector.

  3. Pseudo-elements:

    ::after, ::before, ::selection, etc.
  4. Pseudo-selectors:
    :hover, :focus, :active, etc.
  5. Modifier classes
  6. Nested elements
